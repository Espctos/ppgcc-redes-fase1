\begin{abstract}
This paper presents an experimental comparative analysis between TCP and Reliable UDP
(R-UDP) implementing the Go-Back-N (GBN) sliding window protocol for file transfer
under degraded network conditions. The infrastructure relies on Python sockets running
inside Docker containers (Ubuntu~22.04), with network impairment injected via
\texttt{tc qdisc netem} across three scenarios: 0\%/10\,ms, 10\%/50\,ms, and
20\%/100\,ms of packet loss and latency. A cross-validation methodology confronts
application-level metrics (throughput, latency, retransmissions) against network-level
packet captures via \texttt{tcpdump}. Results show TCP reaching up to 487\,Mbps in
ideal conditions, while R-UDP/GBN achieves only 4.7\,Mbps due to cascaded timeout
retransmissions -- even with 0\% loss. Cross-validation reveals network overhead
between $+52\%$ and $+420\%$ above application bytes, dominated by retransmissions.
\end{abstract}

\begin{resumo}
Este artigo apresenta uma análise experimental comparativa entre TCP e UDP Confiável
(R-UDP) com janela deslizante Go-Back-N (GBN) para transferência de arquivos em
condições de rede degradadas. A infraestrutura utiliza sockets Python em contêineres
Docker (Ubuntu~22.04), com degradação injetada via \texttt{tc qdisc netem} em três
cenários: 0\%/10\,ms, 10\%/50\,ms e 20\%/100\,ms de perda e atraso. Uma metodologia
de validação cruzada confronta métricas de nível de aplicação (vazão, latência,
retransmissões) com capturas de rede via \texttt{tcpdump}. Os resultados mostram o
TCP atingindo até 487\,Mbps em condições ideais, enquanto o R-UDP/GBN alcança apenas
4,7\,Mbps devido a retransmissões em cascata por timeout -- mesmo com 0\% de perda.
A validação cruzada revela overhead de $+52\%$ a $+420\%$ acima dos bytes da
aplicação, dominado pelas retransmissões.
\end{resumo}

% ─────────────────────────────────────────────────────────────
\section{Introdução}
% ─────────────────────────────────────────────────────────────

O desempenho de protocolos de transporte em redes com perdas é tema central em
Redes de Computadores e Sistemas Distribuídos. O TCP oferece entrega confiável e
controle de congestionamento sofisticado via AIMD, mas reage fortemente a perdas,
penalizando a vazão em redes instáveis~\cite{kurose2017}. O UDP, sem estado e sem
garantias, é base para implementações customizadas de confiabilidade como o R-UDP.

Dentre os mecanismos de janela deslizante para R-UDP, o Go-Back-N (GBN) retransmite
toda a janela ao detectar um erro~\cite{tanenbaum2011}, sendo mais simples que o
Selective Repeat ao custo de maior desperdício de banda sob perdas elevadas.

Este trabalho tem os seguintes objetivos: (i)~implementar cliente/servidor TCP e
R-UDP/GBN em Python em contêineres Docker isolados; (ii)~injetar degradação de rede
via \texttt{tc qdisc netem} em três cenários; (iii)~coletar e confrontar métricas de
aplicação e rede; e (iv)~quantificar o overhead real e o impacto das retransmissões
em cascata na vazão e latência observadas.

O artigo está organizado como segue: a Seção~\ref{sec:rel} discute trabalhos
relacionados; a Seção~\ref{sec:met} descreve a metodologia; a
Seção~\ref{sec:res} apresenta os resultados; e a Seção~\ref{sec:conc} conclui.

\section{Trabalhos Relacionados}
\label{sec:rel}
% ─────────────────────────────────────────────────────────────

O QUIC~\cite{iyengar2021} implementa confiabilidade sobre UDP com multiplexação e
reconhecimentos seletivos, superando o GBN por retransmitir apenas o pacote perdido.
O TCP CUBIC~\cite{ha2008}, padrão em kernels Linux, suaviza a degradação de vazão em
enlaces de alta largura de banda e alto atraso, sendo relevante nos Cenários B e C
deste trabalho. A emulação via \texttt{netem} é amplamente utilizada para reprodução
controlada de condições WAN~\cite{hemminger2005}, e sua combinação com \texttt{tcpdump}
para validação cruzada é prática recomendada para fidelidade experimental~\cite{paxson1997}.

\section{Metodologia e Implementação}

\subsection{Arquitetura do Sistema}

O sistema consiste em servidor e cliente Python em contêineres Docker Ubuntu~22.04
interligados por rede \textit{bridge} isolada (\texttt{172.21.0.0/24}). O servidor
escuta nas portas TCP~5003 e UDP~5004. O cabeçalho \textbf{X-Custom-Auth} com valor
\texttt{20261005029:Arthur Sabino Santos} é incluído nos metadados iniciais de cada
transferência, visível nas capturas \texttt{tcpdump}.

\subsection{Protocolo R-UDP com Go-Back-N}

O pacote R-UDP implementado segue o formato:

\begin{center}
\texttt{| seq (4B) | flags (1B) | win\_base (4B) | CRC32 (4B) | payload ($\leq$1400\,B) |}
\end{center}

O cabeçalho ocupa 13~bytes fixos. A integridade é verificada por CRC32. Os flags
definem: \texttt{DATA}~(0x01), \texttt{ACK}~(0x02), \texttt{FIN}~(0x04) e
\texttt{NAK}~(0x08). O GBN opera com janela $W=8$ pacotes e timeout $T=0{,}3\,$s.
O emissor mantém timer único para o pacote base: ao expirar, retransmite toda a janela.
O receptor descarta pacotes fora de ordem e envia NAK imediato.

\begin{lstlisting}[language=Python, caption={Núcleo do Go-Back-N Sender}]
while True:
    while next_seq <= total and next_seq < base + W:
        send_packet(next_seq); next_seq += 1
    if base > total:
        break
    if time.perf_counter() - timer_start > TIMEOUT:
        for s in range(base, next_seq):
            send_packet(s); retransmissions += 1
        timer_start = time.perf_counter()
\end{lstlisting}

\subsection{Coleta e Validação Cruzada}

Para cada experimento foram coletadas duas fontes simultâneas: (1)~\textbf{Aplicação}
-- tempo (\texttt{time.perf\_counter()}), vazão ($\frac{\text{bytes}\times 8}{\text{tempo}\times 10^6}$~Mbps)
e retransmissões; e (2)~\textbf{Rede} -- captura \texttt{.pcap} convertida a CSV
via Scapy, contabilizando bytes e duração. Cada experimento foi repetido 5~vezes por
protocolo por cenário (30~execuções). O overhead é:
$\text{overhead}(\%) = \frac{\text{bytes}_\text{tcpdump} - \text{bytes}_\text{app}}
{\text{bytes}_\text{app}} \times 100$.

O resultado crítico são as \textbf{28.032 retransmissões no Cenário~A} (0\% de perda).
O timeout de 0,3\,s é agressivo para o RTT real de $\sim$20\,ms: expira antes do ACK
retornar sob carga, disparando reenvio de janela inteira desnecessariamente. No
Cenário~C, as 44.047 retransmissões confirmam a cascata do GBN: pacotes retransmitidos
sofrem a mesma perda de 20\%, gerando novas retransmissões. Pela fórmula analítica:
$E[\text{tx/pkt}] = (1-p)^{-W} = (0{,}8)^{-8} \approx 5{,}96$, esperaríamos
$\sim$37.130 retransmissões -- o valor observado (44.047) supera em 18\% pela
contribuição adicional dos timeouts.

arquivo. O overhead TCP no Cenário~A ($+82\%$) reflete pacotes de controle e
retransmissões TCP em maior proporção na interface \textit{bridge} Docker.

% ─────────────────────────────────────────────────────────────
\section{Conclusão}
\label{sec:conc}
% ─────────────────────────────────────────────────────────────

Este trabalho apresentou análise experimental comparativa entre TCP e R-UDP/GBN em
três cenários de degradação de rede com validação cruzada. Resultados principais:
(1)~TCP supera GBN por fator $\sim\!103\times$ no Cenário~A devido a 28.032
retransmissões espúrias por timeout agressivo;
(2)~GBN apresenta latência efetiva $3{,}4\times$ superior ao TCP no Cenário~C;
(3)~retransmissões crescem super-linearmente, superando previsão analítica em 18\%;
(4)~overhead de rede varia de $+52\%$ (TCP, Cenário~B) a $+420\%$ (GBN, Cenário~C).

Como trabalho futuro: ajuste dinâmico do timeout via EWMA para eliminar retransmissões
espúrias no Cenário~A; comparação com Selective Repeat; e validação com SimPy (Fase~2).
